Dolor inicial

El problema del prompting clásico

Funciona para tareas simples. Se rompe cuando queremos repetibilidad, contexto vivo, seguridad y ejecución multi-paso.

$ prompt "actúa como senior dev..."

❌ Copiar contexto manualmente
❌ Explicar reglas cada vez
❌ Repetir criterios de arquitectura
❌ Sin acceso real a Jira, GitHub, DB, logs
❌ Difícil auditar por qué decidió algo
Skills

Una Skill es conocimiento operativo empaquetado.

  • Describe cuándo usarla: el trigger es la descripción.
  • Define cómo trabajar: pasos, criterios, ejemplos y gotchas.
  • Puede incluir scripts: para tareas determinísticas y repetibles.
Principio clave

Una Skill mala no falla por el código. Falla porque el agente no entiende cuándo debe activarla.

Descripción débil

“Ayuda con documentación.”

Descripción fuerte

“Usar cuando haya que crear, revisar o estandarizar documentación técnica para APIs, ADRs, runbooks o handoffs.”

Hooks

Hooks: automatización en los bordes del flujo.

Un hook responde a eventos: antes de ejecutar, después de editar, cuando aparece un error, al cerrar una tarea, al generar un commit.

  • Inyectan contexto sin pedirlo manualmente.
  • Ejecutan validaciones o checks.
  • Aplican políticas del equipo.
MCP

MCP convierte herramientas externas en contexto accionable.

El Model Context Protocol estandariza cómo un agente se conecta a datos, APIs y herramientas: GitHub, Jira, Slack, bases de datos, observabilidad, documentación, CRM, etc.

Sin MCP

Copiar y pegar contexto. El modelo trabaja con snapshots incompletos.

Con MCP

El agente consulta, actúa y valida contra sistemas reales.

Beneficio

Menos fricción, menos contexto manual, más trazabilidad.

Riesgo

Permisos, prompt injection y acciones destructivas mal controladas.

Agentes

El agente es el runtime que coordina el sistema.

No es solo “chat”. Es un proceso iterativo: observa, planifica, actúa, verifica y ajusta.

Vibe coding profesional

No es “programar sin pensar”. Es dirigir sistemas inteligentes.

Intención clara

Definir resultado, restricciones y criterios de aceptación.

Contexto correcto

Dar acceso a repos, issues, docs, logs y convenciones.

Verificación fuerte

Tests, linters, review humano, límites de permisos.

Patrón recomendado

De workflow manual a workflow agentic.

Selecciona un proceso repetido y conviértelo en una pipeline asistida por IA.

Buenas prácticas para el equipo

Reglas operativas

1. Diseñar Skills como APIs

Nombre claro, descripción específica, ejemplos de uso y criterios de salida.

2. Minimizar permisos MCP

Lectura por defecto, escritura explícita, scopes reducidos y secretos fuera del prompt.

3. Hooks para seguridad

Lint, tests, check de secretos, policy checks y bloqueo de acciones peligrosas.

4. PRs auditables

Resumen, cambios clave, riesgos, pruebas ejecutadas y próximos pasos.

Demo sugerida

Construir una mini feature con flujo agentic.

  • Issue: endpoint nuevo o bug pequeño.
  • Skill: estándar de arquitectura/API.
  • MCP: GitHub/Jira/docs/logs.
  • Hooks: test + lint + secret scan.
  • Salida: PR con checklist técnico.

La demo gana cuando el equipo ve que la IA no solo responde: trabaja dentro del sistema.

Arquitectura CLAUDE.md

System vs Project CLAUDE.md

Existen dos niveles que se cargan automáticamente en cada sesión del agente.

  • System CLAUDE.md: reglas globales (empresa / plataforma)
  • Project CLAUDE.md: reglas específicas del repo
  • Se combinan en el system prompt
  • Se ejecutan en cada sesión

Guía seria

¿En qué archivos va cada regla?

No todo debe ir en el mismo CLAUDE.md. La regla práctica es separar por alcance: lo que aplica a toda la organización va en el nivel global; lo que aplica al repositorio va versionado con el proyecto; lo personal queda fuera de Git; y lo específico por carpeta va en reglas path-scoped.

repo/ ├── CLAUDE.md # instrucciones principales del proyecto ├── .claude/ │ ├── CLAUDE.md # alternativa al root CLAUDE.md │ ├── rules/ │ │ ├── testing.md # reglas generales de testing │ │ ├── security.md # reglas de seguridad │ │ ├── api-design.md # reglas para APIs │ │ └── frontend/react.md # reglas por stack o carpeta │ ├── settings.json # configuración compartida del proyecto │ └── settings.local.json # configuración local, no versionada ├── CLAUDE.local.md # preferencias personales del repo └── docs/ └── agent-workflows.md # documentación larga, importable si hace falta
CLAUDE.md PRO

Qué debe contener el archivo principal del proyecto

1. Contexto del repositorio

Explica qué hace el sistema, qué módulos son críticos, dónde vive la lógica principal y qué decisiones arquitectónicas no debe romper el agente.

2. Reglas de comportamiento

Define cómo debe trabajar: preguntar ante ambigüedad, evitar suposiciones, explicar trade-offs y detenerse si falta información crítica.

3. Reglas de cambio

Obliga a hacer cambios mínimos, localizados y auditables. El agente no debe refactorizar, renombrar ni reorganizar código fuera del objetivo.

4. Validación obligatoria

Lista comandos de test, lint, typecheck, build y criterios de aceptación. Cada cambio debe poder verificarse con evidencia concreta.

Principios de ejecución

Convertir principios en reglas operativas

Estos principios deben estar en el CLAUDE.md del proyecto porque aplican a cualquier tarea dentro del repositorio. No son preferencias personales: son estándares de trabajo del equipo.

  • Pensar antes de codear: si la tarea es ambigua, preguntar antes de modificar archivos.
  • Simplicidad primero: preferir la solución más pequeña que cumpla los criterios.
  • Cambios quirúrgicos: tocar solo los archivos necesarios para el objetivo.
  • Orientación a objetivo: transformar instrucciones vagas en criterios verificables.
## Operating Principles

Before editing code:
- Restate the goal in concrete terms.
- Identify ambiguity and ask questions if needed.
- Define acceptance criteria and validation steps.

While editing:
- Make the smallest safe change.
- Do not refactor unrelated code.
- Do not introduce speculative abstractions.

Before finishing:
- Run the relevant checks.
- Summarize changed files, risks, and verification.

Reglas path-scoped

Cuando una regla no aplica a todo el repo

Si una instrucción solo aplica a ciertos archivos, conviene moverla a .claude/rules/ con patrones de path. Esto reduce ruido, ahorra contexto y evita que el agente aplique reglas equivocadas en módulos no relacionados.

Ejemplo: validación de endpoints solo para src/api/**/*.ts, reglas de componentes solo para src/components/**/*.tsx, o convenciones de migraciones solo para db/migrations/**.

---
paths:
  - "src/api/**/*.ts"
  - "tests/api/**/*.test.ts"
---

# API Rules

- Every endpoint must validate input explicitly.
- Use the standard error response shape.
- Add or update tests for invalid inputs.
- Do not change unrelated handlers.
- If the contract is unclear, ask before coding.
Artefacto real

CLAUDE.md + SDD (Specification Driven Development)

Para equipos senior, CLAUDE.md no es documentación, es parte del runtime. Debe integrarse con un enfoque SDD donde cada cambio parte de una especificación verificable antes de escribir código.

El flujo correcto no es “pedir código”, sino:

  • 1. Definir spec clara (inputs, outputs, constraints)
  • 2. Derivar criterios de aceptación
  • 3. Implementar mínimo código para pasar tests
  • 4. Validar contra la spec, no contra intuición
# CLAUDE.md (SDD aligned)

## Before coding
- Restate the specification in concrete terms
- List assumptions explicitly
- Define acceptance criteria (testable)

## During implementation
- Implement the minimal solution
- Do not introduce abstractions unless required by spec
- Limit changes to necessary files

## After implementation
- Validate against spec
- Run tests and checks
- Document deviations or tradeoffs
SDD en práctica

Cómo se ve una spec correcta

Una buena especificación elimina ambigüedad y reduce decisiones implícitas del agente.

# Feature: Email validation

## Input
- string email

## Rules
- must contain "@"
- domain must be valid format
- reject disposable domains

## Output
- boolean valid

## Edge cases
- empty string → false
- null → false

## Tests
- "test@test.com" → true
- "invalid" → false

Anti‑patrones comunes

Comportamientos que deben estar explícitamente prohibidos

Asumir requisitos

El agente implementa edge cases o contratos no pedidos. Corrección: pedir confirmación o definir supuestos antes de tocar código.

Sobre-diseñar

Agrega capas, factories, flags o extensibilidad futura sin necesidad. Corrección: exigir la solución mínima verificable.

Refactor oportunista

Modifica código que funciona porque “se veía mejor”. Corrección: prohibir cambios fuera del objetivo.

Difusión del cambio

Una tarea pequeña termina tocando demasiados módulos. Corrección: pedir justificación antes de ampliar scope.

Validación débil

Entrega sin tests, lint o evidencia. Corrección: checklist obligatorio de verificación.

Resumen insuficiente

No deja claro qué cambió ni cómo revisarlo. Corrección: exigir resumen, riesgos y comandos ejecutados.
